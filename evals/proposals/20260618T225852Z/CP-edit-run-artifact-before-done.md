## Mode (novel — bucket C, model behavioral; harness working correctly)

**False-confidence: declared-done-but-broken.** On `create-chatbot` (a `task_kind="edit"` task), `openai/gpt-5.3-codex` writes `chatbot.py`, verifies it only with `python -m py_compile chatbot.py`, declares completion, and reaches `final_answer`. The harness then runs the strict success probe (`evals/probes/chatbot_smoke.py`: a turn must genuinely round-trip against a mocked client) and correctly buckets the run `probe_failed`. The defect is in the **model**, which substituted a *syntactic* check ("it compiles") for a *functional* one ("it runs a turn"). This is the genuine behavioral residue that ADR-0025/B4's clustering-axis fix was designed to make visible (it previously hid behind a self-contradictory `outcome="success"` failure cluster).

It is the **opposite** of catalogued C1 (won't-conclude / over-persistence → `incomplete`): here the model concludes *too early*, on weak evidence. No existing C entry covers over-confident self-certification of a broken artifact, and no open ADR does either (ADR-0022 is the C1 / investigate-kind lever). Hence `novel=true`.

## Evidence of the precise mechanism

Both failing seeds reached `final_answer` after only `py_compile`:
- **seed3** (`eval_run_20260618T225852Z/...seed3__y21tb0l3`) wrapped `main()` in `argparse.parse_args()`; running it errors `unrecognized arguments` and exits before any chat/completions call. `py_compile` never sees this.
- **seed1/2** (`eval_run_20260618T162508Z`) added a startup `OPENAI_API_KEY` guard; harmless under the real eval (the probe layers `OPENAI_API_KEY=sk-eval-dummy` from the task `[env]` via `score.run_probe`), but emblematic of "guard rails added, program never exercised."

The probe and env are correct (this is **not** a B measurement defect): I confirmed `run_probe` merges `spec.env` over `os.environ`, so the API-key path passes in-eval. The failures are real false-confidence.

## Remediation (prompt_instruction × global → adr_only)

Pull the **same kind-aware mission lever** ADR-0022 uses, but on the `edit` framing (`_KIND_FRAMING["edit"]`, `avatar-harness/avatar/model_client.py:192-196`): instruct the model to **actually run the artifact end-to-end (with `run_command`) and confirm expected behavior before concluding — compiling/linting is not evidence that it works.** Scope strictly to the edit (and test_only) kinds; leave investigate untouched.

Because this is an **always-on prompt rule → `blast_radius = global`**, `ChangeProposal.route()` returns `adr_only`: it lands as an ADR (sibling to ADR-0022), promoted to Accepted only on a measured full-matrix re-run that fixes the gpt-5.3-codex `create-chatbot` `probe_failed` seeds **without** a token/iteration blow-up from over-eager re-running and **without** regressing `modify-existing`, the passing `create-chatbot` seeds, or `investigate-question` (5/5). It does **not** edit any specs/probes/fixtures/verifier/scoring → `touches_grader = false`; the grading surface stays frozen during `validate`.

The narrow, validation-gated framing is what keeps "run it before you call it done" from degrading into noise: it is additive to the edit branch, kind-confined, and the McNemar + agnosticism check is the acceptance gate.
