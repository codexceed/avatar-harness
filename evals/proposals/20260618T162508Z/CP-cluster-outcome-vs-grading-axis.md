## Summary

This cluster is a **construct-validity defect in Workflow A's deterministic clustering spine (bucket B)**, not a harness/model/security failure and not catalog mode A6. The upstream prefilter matched A6 on surface tokens only; the real anomaly is that `evals/cluster.py` labels a genuine *failed* run (`solved=False`) with `outcome="success"`, producing a self-contradictory "create-chatbot success" failure cluster that misleads every downstream consumer (prefilter, judge, human reviewer).

## Root cause

`evals/cluster.py::cluster_failures` (lines 60-71) groups failed rows by `(row.task, row.outcome or "unknown")` and folds `outcome` into the cluster `symptom`. But `outcome` is the harness's **control/terminal axis**, and per `evals/run.py:144-148`, in the **probe-bearing (conversational/non-strict) path** `outcome == "success"` means only *"the agent reached `final_answer`"* â€” it is explicitly decoupled from the grading verdict. The grading truth is `solved = (probe_exit == 0)` (`evals/score.py::is_solved`, option A). Every success-probe task (`create-chatbot`, etc.) therefore emits `outcome="success"` whenever the agent cleanly concludes, regardless of whether the produced artifact works.

So this cluster is real and confirmed in the matrix: in `evals/results/20260618T162508Z.jsonl` the two `create-chatbot` z-ai/glm-5.2 runs (seeds 1 and 2) are `solved=False, outcome="success", probe_exit=1, probe_role="success"` â€” the agent wrote `chatbot.py`, declared the task complete, reached `final_answer` (â†’ `outcome="success"`), but the strict functional probe `chatbot_smoke.py` (which requires a turn to actually round-trip a mocked OpenAI call) **failed**. The deterministic classifier already names this correctly: `evals/classify.py:38-41` returns **`probe_failed`** ("declared done, but the code doesn't work"), and `tests/test_evals.py:360` already pins that exact `("success", False, 1, "probe_failed")` truth.

The two axes are conflated at exactly the seam ARCHITECTURE/CLAUDE.md warns against ("`phase`/`outcome` deliberately kept separate"; "outcome is the terminal-result axis"). `cluster.py` reaches for `outcome` where it should reach for the grading-truth bucket.

## Why it is harmful (not cosmetic)

1. The cluster `symptom` is seeded with the misleading token `success`, so the token-overlap prefilter mis-fired onto A6 (`list_files` directory glob).
2. The label "create-chatbot success" inside a *failure* cluster forces the judge to spend reasoning adjudicating "is this even a failure?" (the `success-run-clustered-as-failure` confusion) instead of diagnosing the underlying `probe_failed` mode â€” a real, valuable signal (false-confidence: declared-done-but-broken) that this defect actively obscures.
3. Distinct failure mechanisms under one task can be merged under a single `outcome` value, instead of split by classifier bucket.

## Fix (code_logic, Layer-1 deterministic spine, no grader touch)

Key and label clusters on the **deterministic grading-truth bucket** (`evals.classify.classify`, threading the row's journal events as `failure_histogram` already does) instead of the harness `outcome`. Build `symptom` from `task + bucket + action tokens`. `touches_grader=false` â€” this edits only `evals/cluster.py` (Workflow A's read-only spine, `$0` eval spend per ADR-0024 principle 1); it does not modify specs/probes/fixtures/verifier/scoring, and must not change any run's grade (`make eval`â†’`diff` should show zero McNemar movement). Blast radius is local; route is `implement`. Per ADR-0024 principle 3 (dedup) and the orchestrator's Mode B verdict, this is a novel construct-validity fix, not a re-diagnosis of A6.

## Note on the underlying agent behavior

The cleaned-up label will surface these create-chatbot runs as `probe_failed` â€” a legitimate, separately-actionable failure mode (the agent self-certifies on syntax-only evidence and never functionally exercises its output). That belongs in its own future proposal once the clustering is no longer lying about it; this proposal's scope is strictly fixing the measurement defect so that signal is visible.
