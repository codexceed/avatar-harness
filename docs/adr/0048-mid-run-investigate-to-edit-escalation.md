# ADR 0048 — Mid-run `investigate → edit` escalation (model-requested + harness-detected); execution admitted in investigation under the net-zero-diff contract

- **Status:** Proposed
- **Date:** 2026-07-11
- **Deciders:** Sarthak Joshi
- **Related:** [ADR-0005](0005-transient-edits-in-investigate.md) (transient edits in investigate — this extends its *enforcement-not-prevention* posture and net-zero-diff contract), [ADR-0046](0046-verification-steers-in-every-mode.md) (the verifier now *steers* on a failed verdict — the mechanism that makes an escalated run repair, and that catches a net-zero-diff violation instead of laundering it), [ADR-0038](0038-model-declared-semi-frozen-verification-contract.md) (the contract an escalated run freezes), [ADR-0007](0007-dynamic-verification-plan-resolution.md) (the freeze boundary escalation reuses). The intent-routing tightening ("actionable failure report → `edit`") is a **separate, complementary** change; escalation is the safety net for when routing still misses. Seams: `avatar/runner.py` (the loop; a shared `_escalate_to_edit`), `avatar/state.py` (`task_kind` mutability + an escalation record), `avatar/tools/base.py` (`phase_admits_tool` — execution in `investigating`), `avatar/tools/commands.py` (`run_command`/`run_tests` phase sets), a new control tool, `avatar/event_types.py` (a `TaskEscalated` event).

## Context

`task_kind` (`edit` / `investigate`, §7) is set once at `TaskState` construction (by the intent router) and is **never mutated** — grep confirms every assignment is a constructor call. The two axes are deliberately separate: `task_kind` selects the verification contract; `phase` (`investigating → editing → verifying`) gates capabilities. But they interact badly for a *misrouted fix*.

A dogfood journal (`tetris_grok3/events/f068a531…jsonl`, task 3) made this concrete. The goal was a pasted traceback — `ModuleNotFoundError` from running the game — an *actionable failure report*. The intent router classified it `investigate` (a mid-build "question"). What followed, over ~43 turns until the human killed it:

- The model made **7 persisted `str_replace` fix edits**. Those rode ADR-0005's transient-edit clause (`admits_transient_edit`: tier-1 mutation is legal in `investigate`, admitted independent of phase), so every edit **succeeded** — the affordance told the model it was making progress.
- It **could not run the code** to check the fix: `run_command`/`run_tests` are phase-gated to `{editing, verifying}` (`_VERIFY_PHASES`), and an `investigate` task never enters those phases (`is_edit_intent` is edit-kinds-only, so transient edits never advance the phase). The tools were never advertised; the model doesn't even know they exist.
- It **could not legitimately keep the edits**: the investigate contract is net-zero-diff (`no_unintended_diff` — the tree must match the pinned baseline at verification). But that only fires at `final_answer`, which the model never reached — with no execution to confirm the fix and no forcing function, it re-read and re-edited the same files in a loop.

The trap has three walls, none of the model's making: **the affordances lied** (edits succeed), **the capability it needed was invisible** (execution isn't advertised in `investigating`), and **there was no lever to reach it** (`task_kind` is immutable, so the model cannot escalate even if it recognized the problem). Critically, the model **cannot self-rescue by calling a tool that is not exposed** — so any fix must supply *both* the missing capability and a harness-owned way to reach it, not merely ask the model to behave differently.

## Decision

Four coordinated changes, so a misrouted fix is *recoverable* instead of terminal — each closing a hole that a naive version would open.

### 1. Execution in investigation — `run_command` only, side effects captured by the baseline diff

`run_command` (only) becomes available in `investigating` (its phase set gains `investigating`). Reproducing and observing a failure — *run the broken command, see what it does* — is core investigation, not editing, and this is precisely the follow-up the `run_command` docstring already flags as deferred ("a true instrument→run→observe→revert loop needs a follow-up decision"). It is safe to admit because `run_command` **already attributes its side effects**: it snapshots `status_paths()` before/after, stages created paths, and returns `files_changed` — so anything it writes flows into the baseline diff and is caught by `no_unintended_diff` at verification (which, under ADR-0046, *steers* rather than laundering). Enforcement-not-prevention, consistent with ADR-0005; the permission gate (tier-3 approval) is unchanged.

`run_tests`/`run_linter` are **not** admitted in `investigating`. Two reasons, both load-bearing: (a) `run_tests` executes *without* the before/after snapshot `run_command` performs, so test-generated artifacts (`__pycache__`, coverage files, generated fixtures) stay untracked and **evade** the baseline diff — breaking the net-zero guarantee; and (b) it resolves its command from config or the *frozen plan*, neither of which exists in a pre-escalation investigate run, so it would only ever return the canned "no test command configured or discovered" failure. Admitting them later requires first moving side-effect accounting into the common command path (so `run_tests` participates in the diff) and defining a non-freezing baseline command resolution — out of scope here (see Alternatives).

### 2. Escalation resolves a *clean* contract from the baseline, through the standard gate — never a live-disk freeze

The critical hole in a naive escalation: the planner's detection tier resolves the contract against the **live** workspace, which by escalation time may contain files the agent itself wrote (a `Makefile` with a `test:` target, a test file, a CI config). Freezing that would let the model **author its own passing contract** — self-certification laundered through escalation, the exact thing §5 forbids. So escalation does **not** freeze whatever is on disk. Instead:

- **Detection resolves against the pinned baseline**, not live disk — so any test/Makefile/config the agent created *during investigation* is invisible to contract detection. Two equivalent implementations (the planner reads the live filesystem today, so this is a real change): resolve detection against the baseline tree (`git show <baseline>:path`), **or** resolve detection *once at run open* — before any transient edit or `run_command` can write — and cache it. **Implemented: cache-at-open** (simpler; sidesteps baseline-tree reads). It runs for **every task kind, not just `investigate`**: because `run_command` is admitted in `investigating` for edit tasks too (§1), an edit task that resolved its plan lazily (at the first edit-intent gate check) would leave the same planted-contract window open on the ordinary edit path — an approved codegen/init command could stage a `Makefile` *before* the first resolution, and detection would freeze the planted file as the contract. Edit tasks pay this resolution at the gate or the freeze anyway, so hoisting it to run open costs nothing and closes the window. (Config-override (tier 0, user-set) and model-*declared* checks are unaffected — those are explicit user/author-and-run signals, not silently-detected planted files.)
- **Escalation hands the task to the standard greenfield path; it does not run it, and does not bypass it.** The existing declaration gate (ADR-0038) fires when an *edit-intent tool* starts an edit. Escalation therefore does **not** need to run `_must_declare_before_edit` → `declare_verification` → smoke-floor (ADR-0014/0047) itself, and deliberately does not: it flips the kind and leaves the task in `investigating` — exactly the state every edit task *starts* in — so the unmodified bootstrap runs that whole sequence on the very next edit-intent call. A greenfield escalation thus gets a real declared contract or the deliverable-scoped floor, never a planted one, via the ordinary path rather than a parallel one. (This is why §3's flip is kind-only: jumping the phase at escalation time is precisely what *would* skip the gate.)

### 3. `task_kind` gains exactly one sanctioned mid-run transition: `investigate → edit`

A shared runner operation `_escalate_to_edit(state)` flips `task_kind → "edit"` — and **nothing else**. It deliberately does *not* advance the phase and does *not* freeze a plan. The task becomes a normal edit task still sitting in `investigating`, so the standard edit-intent bootstrap runs the declaration gate and advances the phase on the next edit (§2): escalation never *jumps* that gate, and there is no second, parallel freeze path to keep safe. The already-made transient edits become the legitimate start of the edit (an edit task's diff is *supposed* to be non-zero).

Escalation is **one-directional**: `edit → investigate` is never offered — you do not demote a change back into an explanation. It is also **once-only**, and the tool is **kind-gated, not merely phase-gated**: an `edit` task also starts in `investigating`, and an escalated task has already become `edit`, so a phase-only rule would advertise "escalate this INVESTIGATION" to tasks that cannot escalate and then no-op after the human paid a tier-3 approval — telling the model an escalation happened when none did. `switch_to_editing` is therefore admitted only while `task_kind == "investigate"` (one rule in `phase_admits_tool`, which the gate and the `ContextBuilder` share, so what is advertised and what will execute cannot drift). `task_kind` is otherwise still immutable; this is the single exception, journaled as a `TaskEscalated` event — with `trigger` recording whether the model asked unprompted (`model`) or the harness's thrash detector nudged it there (`thrash`) — so the transition is legible in replay/eval.

### 4. Two triggers — both consent-gated *proposals*, on concrete thresholds

Escalation to the edit phase is a **mid-run transition out of `investigating`**, so its triggers fire *while the model is still investigating* — never on a concluded investigation. It **changes the user's requested contract** (an investigation promises "leave no diff"; an edit promises "deliver a change"), so it is never silent: *both* triggers are proposals disposed of the same way — **attended → ask the human; unattended → a configured `autonomous_escalation_policy` (default `deny`)**, mirroring ADR-0039's amendment disposition.

- **Model-requested (a).** A new control tool (e.g. `switch_to_editing`) exposed **in `investigating`** — unlike `run_command`, it *must* be reachable from that phase or it is useless. The model calls it whenever it recognizes the goal is a fix; the call is a proposal, disposed of as above.
- **Harness-detected (b) — the thrash net.** The trigger the model *cannot* supply, because it lacks the signal. It must not fire on a *legitimate* investigation that has added instrumentation and not yet reverted, so the threshold is concrete, not "substantive divergence": the existing repeat detector fires **≥ `escalation_thrash_repeats` times (default 3)** *while* a non-empty baseline diff persists across those repeats. This catches the `tetris_grok3` spiral before it burns a session, without tripping on a single not-yet-reverted print.

Both triggers converge on the same `_escalate_to_edit` (§3).

**The investigation-*ended* case is deliberately not an escalation trigger.** When the model proposes `final_answer` in an investigate task with a non-empty baseline diff, that is the *end* of the investigation, not a mid-run state — reframing a concluded investigation as an edit would be presumptuous, and a leftover diff there is as likely forgotten instrumentation. So it is handled by the existing contract, not by escalation: `no_unintended_diff` fails → ADR-0046 steers → the default is **revert to baseline**, and the steer feedback carries a **nudge** surfacing `switch_to_editing`. Escalation remains reachable at that point *only via the model's own request* (a) in response to the nudge — never a harness auto-flip of a completed investigation.

### 5. Legibility

The investigate contract is stated in-context: *"You can run code to observe behavior, but an investigation must leave the tree at its baseline (net-zero diff). If you are fixing something, switch to an edit task — that keeps your changes, lets you run and verify them, and holds them to a real contract."* This names the limitation **and** points at the now-reachable lever — the description is only safe because (2)+(3a) make it actionable.

## Alternatives considered

- **Model-requested escalation only.** Rejected: a tool the model must call cannot rescue a model that has no signal it is stuck — its edits succeed, so it perceives no problem, and it cannot invoke a capability it was never shown. The `tetris_grok3` run is the proof. Kept as *one* trigger, never the sole one.
- **Harness-*auto* escalation (act without consent) on a diff/thrash signal.** Rejected: an investigation legitimately adds instrumentation, runs it, and reverts — a not-yet-reverted diff is the *normal* mid-state, not a fix. Auto-escalating on it would silently convert "leave no diff" into "deliver a change" against the user's intent. Hence §4: the thrash detector is a *proposal* (attended asks; unattended per `autonomous_escalation_policy`, default `deny`), on a concrete threshold (≥N repeats with a persistent diff), never an auto-flip.
- **Treat `final_answer`-with-diff as an escalation trigger.** Rejected: that is the *end* of the investigation, not a mid-run state — escalation to the edit phase is a transition *out of* `investigating`, which a concluded investigation is past. A leftover diff there is handled by the existing contract (`no_unintended_diff` fails → ADR-0046 steers → revert), with a nudge that lets the model *request* escalation if it really meant to fix; the harness never auto-reclassifies a completed investigation.
- **Freeze the contract from the live workspace at escalation** (the naive `_escalate_to_edit`). Rejected: it lets the agent *plant* its own contract — write a `Makefile`/test/CI file during investigation, then have detection freeze it as the passing rubric. Self-certification laundered through escalation. Hence §2: detection resolves against the pinned baseline, and escalation runs the standard declaration/floor gate instead of trusting disk.
- **Admit `run_tests`/`run_linter` in `investigating` alongside `run_command`.** Rejected for now: `run_tests` executes without `run_command`'s before/after side-effect accounting, so test artifacts evade the baseline diff (breaking net-zero), and it has no resolved command pre-escalation (canned failure). Admitting it needs prior work — move side-effect accounting into the shared command path, and a non-freezing baseline command resolution — tracked as a follow-up, not bundled here.
- **Keep `run_command` out of `investigating` entirely.** Rejected: reproducing a failure is the essence of investigation, forcing the model to edit-blind is half the thrash, and `run_command` *already* attributes+stages its side effects so the net-zero contract still holds.
- **Make `investigate` strictly read-only (remove ADR-0005 transient edits).** Rejected: loses legitimate instrumentation (add a print, run, observe) — which change (1) now makes actually useful — and still leaves a misrouted fix with no recovery path.
- **Fix intent routing only; no escalation.** Rejected as *sufficient*: routing is never perfect, and a miss would remain terminal (thrash-until-killed). Routing is still worth doing — it makes escalation rare rather than the primary path — but it is the complement, not the substitute.
- **A generic bidirectional `set_task_kind`.** Rejected: `edit → investigate` has no valid use (you do not turn a committed change into an explanation), and a general mutation invites incoherent states. One sanctioned direction only.

## Consequences

- **`task_kind` is no longer immutable.** §7 must document the one sanctioned transition; `TaskState` carries an escalation record and emits `TaskEscalated`; artifact/eval consumers that keyed on "kind is fixed for the run" must read the final kind. Verification, context framing, and the diff contract all read the *current* kind, so they follow the flip naturally.
- **Investigations can execute (`run_command`).** A diagnosis goal can reproduce a bug and confirm a hypothesis; `run_command`'s existing side-effect accounting keeps its outputs inside the baseline diff, and `no_unintended_diff` (with ADR-0046 steering) is the enforcement that keeps an investigation from silently leaving changes. `run_tests`/`run_linter` stay editing/verifying-only pending the shared-accounting follow-up.
- **The contract can't be self-authored via escalation.** Detection reads the pinned baseline and escalation runs the standard declaration/floor gate, so a file the agent wrote during investigation can never become its own passing rubric — the §5 no-self-certification invariant survives the new transition.
- **Escalation is consented, never silent.** Both triggers are proposals; a legitimate investigation that touches the tree and reverts is never force-converted to a delivered change. New config: `autonomous_escalation_policy` (default `deny`) and `escalation_thrash_repeats` (default 3).
- **A misrouted fix is recoverable.** Escalation binds a contract, exposes run/verify natively, and — via ADR-0046 — gains a forcing function (steer to pass, or block at exhaustion) instead of spinning until a human intervenes.
- **Interplay with ADR-0046 is load-bearing.** Escalation is only valuable because the edit path now *steers*; without ADR-0046 an escalated run would inherit the old advisory hole. The two ADRs compose: 0046 makes verification steer; 0048 makes a misrouted run *reach* verification.
- **The intent router still matters.** Escalation is the net, not the happy path — starting a fix as `edit` is a better UX than starting `investigate` and switching. The routing tightening proceeds separately.
- **New surfaces to test:** `run_command` admitted in `investigating` (and its outputs appearing in the baseline diff there); `run_tests`/`run_linter` still refused in `investigating`; baseline-resolved detection ignoring an agent-planted `Makefile`/test; escalation running the declaration/floor gate; `_escalate_to_edit` (kind/phase/plan-freeze, edits preserved); the `switch_to_editing` tool (exposed in `investigating`, proposal-gated); the thrash detector firing on ≥N repeats-with-persistent-diff and *not* on a single not-yet-reverted edit; `final_answer`-with-diff in an investigate task steering to *revert* (not escalating) while surfacing the `switch_to_editing` nudge; the `autonomous_escalation_policy` dispositions; the `TaskEscalated` event round-trip.
