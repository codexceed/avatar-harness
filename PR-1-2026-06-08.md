## Change Review

### `src/avatar_harness/workspace.py`

| Line(s) | Severity | Feedback |
|----------|----------|----------|
| `_parse_patch_targets` | DO | Path confinement only inspects `--- `/`+++ ` headers. A git diff using rename headers (`rename from`/`rename to`) or a `diff --git a/x b/y` with no `---`/`+++` body (pure rename / mode change) yields targets the parser misses, so both the permission gate's confinement check and `apply_patch`'s own `_resolve` loop can miss a rename *destination* outside the root. `git apply` honors those headers. Either parse rename/`diff --git` headers, or reject any diff whose `diff --git` lines reference paths the `---`/`+++` parse did not cover. |
| `apply_patch` | CONSIDER | The `git apply --check` then `git apply` two-step re-runs git twice over the same diff; a single `git apply` with a try/rollback would be cheaper, but the check-first form is clearer and the cost is negligible. Keep as is unless profiling says otherwise. |
| `run` | CONSIDER | `shlex.split(command)` plus `subprocess.run` without a shell is the right call for confinement, but it silently cannot run shell constructs (pipes, `&&`, globs). Given the default `test_command`/`lint_command` are plain argv, fine — worth a one-line docstring note that shell metacharacters are not honored. |

### `src/avatar_harness/verifier.py`

| Line(s) | Severity | Feedback |
|----------|----------|----------|
| `_command_check` / `_NO_TESTS_EXIT` | TRY | Exit codes 4 (target-not-found) and 5 (no-tests) are hard-coded pytest conventions, but `test_command` is arbitrary config. If a user sets a non-pytest command, a 4/5 exit is silently misclassified as a tolerated skip and could let a vacuous verification pass — the exact failure mode CLAUDE.md §12 warns against ("never vacuously on skipped checks"). The PR notes this in the ledger; at minimum couple the exit-code semantics to the command (or gate them behind a `pytest`-detection) so the coupling is explicit in code, not just prose. |
| `_no_secrets` / `_scan_secrets` | CONSIDER | The marker list (`AKIA`, `-----BEGIN`, `PLACEHOLDER`, `<placeholder>`) will false-positive on legitimate diffs that *discuss* those tokens (docs, tests, this very file). Acceptable for an MVP guard, but consider scoping it or making it advisory rather than a hard required-fail. |

## Testing Assessment

Coverage is strong and the suite is green (**74 passed** locally, matching the claim). The new tests are well-targeted: atomic multi-file apply, path-escape rejection, stale-patch no-op, timeout, clean-start assertion, all three §12 verifier criteria (including the zero-positive-signal and disallowed-skip cases), the gate tiers, and an end-to-end edit-to-verified-success runner test. The "command ran but failed is data" distinction is explicitly tested. This is high-quality TDD work.

Gaps, in priority order:

- **DO** — No test asserts `state.commands_run` is populated after a `run_tests`/`run_linter` call (because it never is). A test here would have caught the runner gap and the empty-artifact consequence.
- **DO** — No test covers the dirty-repo path through `run_agent`/`main`. `test_workspace_open_rejects_dirty_unless_allowed` covers `Workspace` directly, but nothing exercises the CLI's behavior on a dirty tree, so the uncaught-crash regression is invisible to the suite.
- **TRY** — No test for a rename-header diff against path confinement (the `_parse_patch_targets` gap). Add a diff with `rename to ../outside` and assert the gate/`apply_patch` refuses it.
- **TRY** — `ArtifactManager` is tested in isolation but there is no integration test asserting the CLI/runner actually produces an artifact for a finished run.
- **CONSIDER** — The verifier's pytest-exit-code coupling (4/5) is tested only with synthetic `python -c "sys.exit(N)"`. Given the input space is "any configured command," a note or a guard test for a non-pytest command exiting 4/5 would harden the vacuous-pass risk.

## Summary

**Approval Status:** CHANGES REQUESTED

This is a careful, invariant-respecting PR with excellent tests and documentation discipline — the core Phase 2 mechanics (atomic patch, harness-owned verifier, synchronous gate, clean-start baseline) are well-built and faithful to the design spec. It is held back by a small set of concrete gaps: a CLI dirty-repo crash with no acknowledgement path, `state.commands_run` never being recorded (which leaves the artifact's command list inert), an `ArtifactManager` that is built but unwired, and a path-confinement hole on rename-header diffs. None are deep, but several are user-visible or security-relevant.

**Findings:** 4 DO, 6 TRY, 6 CONSIDER

### Critical Items (DO)
- `workspace.py:_parse_patch_targets` — rename/`diff --git`-only headers bypass tier-1 path confinement; parse them or reject diffs they don't cover.
- `runner.py:_apply_tool_result` — command tools never populate `state.commands_run`; artifact command list and investigate `inspected` signal are blind to commands run.
- `cli.py:run_agent` — default `Workspace(".")` with `allow_dirty=False` crashes (uncaught `DirtyWorkspaceError`) on any dirty repo; add an `--allow-dirty` flag/setting and catch the error.
- `tests` — missing tests for `commands_run` population and for the CLI dirty-repo path; either would have caught the above.

### Suggested Items (TRY)
- `verifier.py:_command_check` — pytest exit-code (4/5) semantics applied to an arbitrary configured command risks a vacuous skip-pass; couple the codes to the command.
- `permission.py` — imports private `workspace._parse_patch_targets`; promote to a public helper.
- `artifact.py` — `ArtifactManager` is built and tested but never wired into runner/CLI; wire it or document it as deferred.
- `tools/__init__.py:default_registry` — un-advanced `phase` means a real model is never shown `apply_patch`; the closed edit loop is scripted-only until phase advancement lands.
- `ARCHITECTURE.md` — "[Implemented] whole engine" overstates the demonstrable state given the carried gaps; soften.
- `tests` — add a rename-diff confinement test and a runner→artifact integration test.

### Optional Items (CONSIDER)
- `workspace.py:apply_patch` — double `git apply` invocation; minor.
- `workspace.py:run` — document that shell metacharacters are not honored.
- `verifier.py:_scan_secrets` — marker scan false-positives on diffs that discuss the markers.
- `runner.py:L124` — the double-None permission/tool dance reads awkwardly.
- `artifact.py:L42` — `str(state.outcome)` would stringify `None`; use `state.outcome or "unknown"` defensively.
- `model_client.py` — fresh-`DecisionParseError` re-raise is clean; no change needed.
