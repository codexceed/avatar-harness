# DECISIONS — design decision log

**The running record of *why* the build is shaped the way it is.** Each entry is a dated decision: the choice made, the alternatives weighed, the trade-off accepted. `PROGRESS.md` tracks *how far* the build has gotten (checklists); `HARNESS_DESIGN.md` is the *what/why* spec; `ARCHITECTURE.md` is the current system map. This file is the chronological ledger of decisions taken along the way.

> **When to add an entry:** whenever a **major design decision** is made — a chosen approach, a rejected alternative, a scope cut, a non-obvious trade-off, or a load-bearing clarification. Append a dated bullet (`- **YYYY-MM-DD** — …`) and keep the *why* and the alternatives, not just the *what*. Routine implementation that merely follows an existing decision needs no entry.

- **2026-06-09** — **3.2b — `@path` grounding** (tail regrouped into smaller slices: 3.2b `@path` · 3.2c plan mode · 3.2d verify-authority · 3.2e CLI launch). A goal mentioning `@path/to/file` seeds that file as initial `kind="grounding"` evidence on the fresh `TaskState` — the same explicit-seeding mechanism as history (`add_feedback`), so the model sees it as context without transcript bleed and invariant #1 holds. **Read through the `Workspace`, not `open()`** — the load-bearing decision: grounding goes through the same chokepoint as every other read, so the sensitive-path denylist + path confinement apply for free (`@.env` is refused exactly like a `read_file` on it; a test asserts the secret never enters state). A refused/missing/out-of-root path (`SensitivePathError`/`PathOutsideWorkspaceError`/`OSError`) becomes a short note rather than a crash. The `@token` is left in the goal text (the model still sees the reference); content is capped per file (a hint, not a dump); a throwaway `Workspace(allow_dirty=True)` is used since grounding is a read-only inspection. 5 tests; 221/221 green.
- **2026-06-09** — **3.2a — meta commands, the first slice of the 3.2 tail** (decomposed into 3.2a meta commands · 3.2b `@path`+plan mode · 3.2c verify-authority+CLI launch). - **2026-06-09** — **Lane 2c — cockpit modals (approval / diff / plan), completing Lane 2.** Three `ModalScreen`s, each dismissing a small typed result the caller routes — keeping the modal a pure view that *returns a decision*, not a thing that reaches into the engine. **`ApprovalModal`** returns `ApprovalChoice(allow, remember)` via `[y]`/`[a]`/`[d]` key bindings (`[a]`→`remember=True`, the PR-#10 scoped grant; `[v]` toggles detail); `CockpitApp` auto-pops it when an `ApprovalRequested` arrives on the stream and, on dismiss, calls `session.resolve_approval` in a worker — so the **event announces and the modal decides** (§13), the two planes stay separate, and the event consumer never blocks on the modal. **`DiffModal`** is read-only (escape/enter → `dismiss(None)`). **`PlanModal`** uses an editable `TextArea` + **Button**s (not key bindings — a focused `TextArea` would swallow letter keys, so approve/revise are clicks) and returns `PlanChoice(approved, text)` carrying the current editable text, so "revise" genuinely returns the edit. Headless tests drive a tiny host `App` that `push_screen_wait`s the modal and captures the result — asserting the **result contract**, not pixels. The plan-mode *flow* (read-only plan task → approve → seed the edit task) and the `/diff`/`/plan` meta-commands that trigger these modals are the 3.2 tail; 2c builds the widgets + contracts. 7 tests; 208/208 green.

- **2026-06-09** — **Lane 2 decomposed; Lane 2a — multi-turn `SessionState` + `ReplSession`** built first (pure logic, no Textual). Lane 2a decisions: **(1) The multi-turn scope is a new `SessionState` (pydantic data per §23.1) + a thin `ReplSession` driver; the single-task `Session` is reused per goal, not replaced** — §23.2's "one code path," so batch is the degenerate one-`submit` session (verified by a test asserting `submit` == `arun`). **(2) Standing grants moved up to session scope.** PR #10's `ApprovalGrant` lived on the per-task `Session`; a REPL needs `[a] always` to persist across goals, so `SessionState.grants` is the owner and `Session` gained a `grants=` seam that shares the list **by reference** — a grant remembered in task 1 auto-allows in task 2 (the gate still asks every time; the session answers from the shared list). **(3) Mode = visible heuristic default + explicit override** (ADR-0002 D3, no hidden classifier): `default_mode` routes a first-word imperative → `edit`, else `investigate`; `set_mode` overrides; the `/mode` *meta-command* that calls it is the 3.2 tail. **(4) History seeds the next task as explicit initial evidence** (`add_feedback(kind="history")`), so a fresh per-goal `TaskState` (invariant #1) still knows the conversation — but **in-session only** (no cross-session persistence; deferred) and curated through the evidence channel, never raw transcript bleed (the structured state stays the source of truth). **(5) `ReplSession.record(state)` takes only the terminal state** (dropped the redundant `session` arg — `TaskState` carries `task_id`/`final_answer`/`outcome`); `start`/`record` are the lower-level pair the cockpit drives (observe `events()` between them), `submit` is the run-to-completion convenience. `ReplSession`/`SessionState`/`Turn` exported. 8 tests; 194/194 green.
- **2026-06-09** — **Lane 1 — bounded `EventBus` + privileged write-ahead `JsonlEventJournal`** (ADR-0001), built sequentially. Four scope calls, all toward Principle C (thin): **(1) The journal is bus-internal and sync-flushed per event, lossless; the bus publish path stays non-blocking.** Every published event is appended to the journal *before* the fan-out — so the journal is complete even when a subscriber sheds. The ADR's *awaited* journal + checkpoint/resume is **deferred to 3.3**; "route `arun()` through `await bus.publish()`" is realized as the existing sync publish path writing the journal, which means **zero `runner.py` edits** (the team-prelude's stated goal). **(2) Bounded queues are *soft* bounds.** A subscriber's cap sheds only droppable `*_update` events (today just `model_update`); lifecycle/control events are always enqueued and may exceed the cap — that is what "never drop lifecycle" means mechanically. Drops are visible as `event_id` gaps at the subscriber while the global sequence (and journal) stays contiguous. **(3) Drop-newest, not coalesce.** Shedding the incoming update at the cap is the thinnest correct behavior for "sheds low-priority under pressure"; coalescing (replace the last pending delta) is deferred until a renderer actually needs it. **(4) One hardcoded drop policy, no `DropPolicy` enum/param.** `max_queue` is the only knob `subscribe()` gained; the second policy (OTel's "drop oldest") doesn't exist yet, so per rule-of-three it isn't abstracted. **File split executed now** (`EventBus`→`bus.py`, new `journal.py`): the bounded-bus + journal growth is the friction that justifies the split flagged in the prior entry — done because the work needed it, not as team scaffolding. `JsonlEventJournal` kept distinct from the sync-`Emitter` `EventLog` (engine commit log vs emitter subscriber; the journal holds its handle open and flushes per event — the write-ahead property `EventLog`'s reopen-per-call lacks). `Session` gained an optional `journal=`; a journal-less session is byte-for-byte the 3.0 foundation behavior. 10 tests; 186/186 green.
- **2026-06-09** — **Prefix-scoped `ApprovalGrant` (`[a] always`) lives in `Session`, not `PermissionPolicy`.** The standing-approval affordance for tier-3 `run_command` is a *control-plane* convenience layered on the approval announcement: the harness gate (`PermissionPolicy.check`) still returns `ask` for **every** tier-3 call — it stays pure and harness-owned (invariant #4) — and the `Session` may *answer* that ask from a remembered grant instead of prompting the human. Consequence: the runner's `_approved`/`request_approval` path is **unchanged**; the whole feature is `session.py` + one additive event field. Rejected putting grants in the policy (would make the harness gate stateful/session-aware and couple it to UI memory) and threading the tier through the `ApprovalController` protocol (the Session reads the tier from `runner.registry` instead, leaving the protocol frozen). **Three sub-decisions (maintainer-signed-off):** (1) **Grant unit = the command's program (`argv[0]`)**, mirroring Claude Code's `Bash(pytest:*)` — approving `pytest -q` grants `pytest …`, a different program re-prompts; useful (the model varies args) and safe (breadth is bounded by the program). Rejected full-argv-prefix (too narrow — the model rarely repeats an identical command). A non-command tier-3 tool grants on its name. (2) **Auto-allows stay observable** via an additive `ApprovalResolved.via: Literal["human","grant"]="human"` (schema stays v1 — backward-compatible); a granted call emits **no** `ApprovalRequested` (that event means "a human must decide"; a grant skips the human) and is recorded as `ApprovalResolved(via="grant")` (invariant #5). (3) **Never global, never tier-4, never cross-session** — an empty program prefix matches nothing (no match-everything grant), the match predicate refuses tier ≥ 4 (destructive always re-prompts), and grants live on the `Session` instance (no process-global leak; a fresh session starts empty). `ApprovalGrant` exported on the public two-plane surface. 8 tests; 176/176 green; `make check` clean.

- **2026-06-09** — **Advertised-tools must mirror admitted-tools** (PR #6 review fix). The phase-advance bootstrap (2.6) admits the edit-intent tool (`apply_patch`, tier 1) from `investigating` on edit tasks — but the `ContextBuilder` advertised only `active_for_phase(state.phase)`, which excludes it. - **2026-06-08** — Sensitive-path matching **normalized to one matcher**. Review found two engines interpreting the same `sensitive_path_globs`: `path_is_sensitive` (component `fnmatch`) at the gate + workspace, vs ripgrep's gitignore `-g` globs in `search_repo` — divergent on slash patterns (`fnmatch`'s `*` crosses `/`, gitignore's doesn't). - **2026-06-08** — Phase 2.5 opened from a live dogfood (`events/ff24fa3c…jsonl`, a "rich chat app" investigate run that produced functional-but-UI-buggy code). - **2026-06-08** — Session grouping made explicit (REPL precursor; §23).
