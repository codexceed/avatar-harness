# DECISIONS — design decision log

**The running record of *why* the build is shaped the way it is.** Each entry is a dated decision: the choice made, the alternatives weighed, the trade-off accepted. `PROGRESS.md` tracks *how far* the build has gotten (checklists); `HARNESS_DESIGN.md` is the *what/why* spec; `ARCHITECTURE.md` is the current system map. This file is the chronological ledger of decisions taken along the way.

> **When to add an entry:** whenever a **major design decision** is made — a chosen approach, a rejected alternative, a scope cut, a non-obvious trade-off, or a load-bearing clarification. Append a dated bullet (`- **YYYY-MM-DD** — …`) and keep the *why* and the alternatives, not just the *what*. Routine implementation that merely follows an existing decision needs no entry.

- **2026-06-10** — **The TUI is a consumer, not a core feature: the cockpit launch inverted out of `cli.py` into its own `jo-cli` entry point.** The core CLI imported `avatar_harness.tui` (`load_cockpit`) and owned `--interactive`/`--auto` — the engine's shell knew one specific consumer, while the harness is meant as an independent core under many (TUIs, eval drivers, autonomous wrappers). Maintainer call: a sensible agentic stack, with the cockpit named **jo** (`jo-cli`).
  - **The inversion:** `jo-cli` (`avatar_harness.tui.cli:main`, a second `[project.scripts]` entry) absorbs the cockpit launch + `--auto`/`--log`/`--allow-dirty`; `avatar-harness` is the pure batch shell and rejects the old flags. Import direction is now strictly TUI → core, enforced by test (`import avatar_harness.cli` pulls in neither `textual` nor `avatar_harness.tui`).
  - **Shared plumbing moved down, not sideways:** the per-session journal-layout helpers (`resolve_log_path`, `update_latest_pointer`) moved from `cli.py` to `journal.py`, so both shells consume one layout and the TUI never imports the batch shell.
  - *Rejected:* entry-point plugin discovery for UIs (a framework before a second consumer exists — Principle C). *Deferred:* the full separate-distribution split (`jo-cli` as its own package) — audited 2026-06-10 as mechanical (~hours, zero blockers: the TUI already consumes only the exported surface); take it when a release cadence demands it.

- **2026-06-10** — **Mode routing is LLM-classified — visible and correctable (revises ADR-0002 D3)** (dogfood `events/04849a5a…jsonl`). The first-word heuristic routed "Now make the UI richer…" to `investigate`; with mutation structurally impossible for that kind, the run burned all 50 turns on reads. D3's recorded objection was *hiddenness*, not LLM-ness — the rejected design classified silently. This keeps the principle and replaces the brittle part.
  - **One cheap, schema-constrained call** (`ModeClassifier`, new `intent.py`): a forced `set_task_mode(kind enum)` function call on `AVATAR_CLASSIFIER_MODEL` (default `openai/gpt-5-nano`, ≈$0.00003/goal — three orders of magnitude under one wasted turn), fed the prompt + recent conversation so follow-ups classify from context.
  - **Precedence: `/mode` override → classifier → hardened heuristic.** The heuristic now skips leading filler ("now/please/also/can you…") and routes question-openers to `investigate` — it remains the offline/failure fallback, and `classify()` returns `None` on any failure (routing can degrade, never block a goal).
  - **Visible, never silent:** the cockpit announces `▶ mode: <kind> (<source>) — /mode to change` before the run and updates the status bar; `last_mode_source` records how the verdict was reached. The verdict is memoized per prompt (`start()` re-resolves; one network call per goal), and resolution happens off-loop (`to_thread`) so the TUI never freezes.
  - **Tests are offline by design:** `classifier_model` defaults ON, so a conftest autouse fixture clears `AVATAR_CLASSIFIER_MODEL` suite-wide — a bare `ReplSession` in a test must never call a real endpoint (caught when the first full-suite run spent 343s in provider retries).
  - Agent-initiated mid-run escalation deferred — only if classification still misroutes in practice. 13 tests; 289/289 green.
- **2026-06-10** — **Context compaction budgets made realistic, configurable, and loud** (dogfood `events/63bced3f…jsonl`). A follow-up goal burned its whole 50-turn budget with 42 re-reads of `scripts/chatbot.py` and zero edit attempts: the Phase-2.5 compaction silently cut every evidence detail at 1,500 chars, so the model saw a 3,548-char file that appeared to end mid-function — and *modifying* a file (a diff's exact context lines, or a full `write_file(overwrite)` rewrite) requires seeing all of it. Creation worked; modification was structurally unwinnable.
  - **Truncation is never silent:** a cut detail gains `… [truncated: shown/total chars shown]`. Wording is deliberately **neutral** — detail also comes from commands/search/verifier (all `kind="tool_result"`/`verification`), where tool-specific advice like "re-read with line_range" would itself mislead (review fix).
  - **Degrade is loud too:** an over-budget item collapsed to summary-only appends `[detail elided: N chars]`, so the model can distinguish "summary-only result" from "content aged out". Evidence that never had detail stays unmarked.
  - **Budgets are `HarnessConfig` fields** (`AVATAR_CONTEXT_MAX_DETAIL_CHARS` / `AVATAR_CONTEXT_DETAIL_CHAR_BUDGET`), threaded into the default `ContextBuilder` by the facade; an injected builder keeps its own budgets (Principle-A seam unchanged). The old constants were invisible and unreachable.
  - **Defaults sized for the job:** 16k chars/item (an ordinary source file fits whole) / 48k total — ≈12k tokens against the 100k-token `max_context_tokens` guard, which still bounds the whole packet. Both defaults live once in `config.py` (`DEFAULT_CONTEXT_*`) so config and builder can't drift (review fix).
  - Marker chars aren't charged to the budget and the pre-spend check can overshoot by one item — accepted and commented; the packet-level guard is the hard bound.
  - 8 tests (incl. verifier-pin truncation, degrade elision, env-isolated defaults); 284/284 green.
- **2026-06-09** — **`apply_patch` teaches the unified-diff format instead of accepting the `*** Begin Patch` dialect** (dogfood `events/041fde1e…jsonl`). OpenAI-family models emitted their in-house patch dialect in two consecutive runs; the generic "no file targets found in diff" error corrected nothing, and one run burned its remaining budget on blind same-dialect retries → `incomplete`.
  - **Recognize the dialect, answer with the fix:** the handler detects the `*** Begin Patch` markers and returns a model-correctable error that names what it saw, spells out the expected format (`--- a/<path>` / `+++ b/<path>` / `@@` hunks, `/dev/null` for create/delete), and offers the `write_file(overwrite=true)` fallback that rescued the prior run.
  - **The tool description specifies the format markers** — under native tool-calling (ADR-0003 A) it is the literal function-schema text the model sees, so it must teach, not gesture.
  - **Translating the dialect deferred** (rule of three): Begin-Patch carries no line numbers, so a converter needs context-matching — take it on only if a third run still trips.
  - 2 tests; 271/271 green.
- **2026-06-09** — **Multi-turn §15: session-owned dirt never blocks a follow-up goal; a failed goal never kills the cockpit** (dogfood crash, `events/2110f1e1…jsonl`). Goal 1 succeeded (`write_file` staged `scripts/chatbot.py`); goal 2 died at the clean-start check *on the session's own output*, and the `DirtyWorkspaceError` escaped the Textual worker, tearing down the TUI mid-conversation.
  - **The clean-start check protects the sitting's FIRST goal only.** Its §15 purpose is keeping pre-existing user changes out of the diff baseline; once the first per-goal workspace opens cleanly, later uncommitted changes are the session's own work product — `ReplSession` tracks `_tree_claimed` and opens subsequent goals `allow_dirty`. The flag flips only on a successful open, so a refused first goal stays refused on retry. *Rejected:* `allow_dirty=True` for all REPL goals (loses the first-goal protection). The diff baseline stays consistent regardless (it pins HEAD; the harness never commits).
  - **`--allow-dirty` is honored interactively** (`ReplSession(allow_dirty=…)`); previously it silently applied only to batch.
  - **The cockpit survives a failed goal:** `_run_goal` catches `DirtyWorkspaceError` (rendered with the commit/stash/`--allow-dirty` hint) and any other per-goal exception as a `✗` transcript line, re-enabling the prompt — an interactive shell degrades to an error message, never crashes.
  - 5 tests; 274/274 green.
- **2026-06-09** — **`write_file` — first-class file creation (ADR-0003 Option B, implemented).** Creation gets a plain-content transport; the diff costume comes off the no-anchor case.

  - **Modification stays diff-anchored:** an existing target raises `FileExistsError` → a model-correctable result that names `apply_patch`, unless `overwrite=true` is explicit — the clean-apply staleness invariant can't be bypassed casually.

- **2026-06-09** — **Native tool-calling is the default decision transport (ADR-0003 Option A, accepted & implemented).** Maintainer took A *first* (the ADR recommended B-then-A): decisions ride the provider's function-calling channel instead of a hand-written JSON envelope. `build_tool_schemas` sends every phase-admitted tool as a function (its real pydantic `input_schema`) plus `final_answer`/`ask_user` — every decision shape is provider-validated, so a 200-line patch can't die in JSON escaping.
  - **Compatibility is automatic, not configured:** a reply with no `tool_calls` falls through to the legacy `parse_decision` on its content; `AVATAR_NATIVE_TOOL_CALLS=false` additionally forces the legacy protocol verbatim (escape hatch; default on).

  - **Retries keep the §18 invariants:** malformed arguments retried in-conversation with valid `tool_call_id` pairing, landing on the `retry_trace` — composes with the observability fix.
  - **The native system prompt drops the JSON-envelope contract** (it conflicted with tool-calling) while staying kind-aware; the legacy prompt is byte-identical for the fallback path.
  - **First tool call wins** when a model emits several (§6: one action per turn). `ModelClient` ABC, runner, tools, verifier untouched (adapter-internal, Principle A). One Phase-1 test assertion moved with the contract (tool advertisement now asserted in `tools=`, not prompt prose).

- **2026-06-09** — **Decision-parse failures are observable end-to-end; the cockpit renders the verifier's verdict** (dogfood post-mortem, `events/0ad6c3fe…jsonl`). The model's `apply_patch` attempts died as `DecisionParseError`s *inside* the client's retry loop — no event, no evidence, no ledger entry — so it looped 24 turns on reads, hallucinated "I already applied the patch", and the cockpit showed `■ success` even though the verifier failed `diff_present` (conversational mode delivered the reply; the promised ✓/⚠ badge had never been implemented).
  - **Annotate, don't swallow:** the in-client retry loop stays (it preserves the raw malformed reply in-conversation — the best correction signal) but annotates the surviving decision with a `retry_trace`; the **runner** turns each note into `decision_error` evidence (the model sees its own failed attempts) and a typed `DecisionError` event (`recovered=True/False`; the exhausted path publishes to the bus too). *Rejected:* threading an event sink into `ModelClient` — the client stays a pure adapter.
  - **`retry_trace` is a harness-owned channel:** `parse_decision` clears it, so a model can't plant fake history.
  - **The retry nudge asks for the SAME intended action**, not "a valid decision" — the old wording invited the downgrade-to-a-read cope that fueled the loop.
  - **The cockpit renders the verdict** (`✓/⚠ verification …` line + a `verify:` status-bar badge) and `↩ malformed decision` lines — in conversational mode `outcome` alone reads "success", so without the badge the human "terminal authority" decided blind.

Goals stay distinguishable by per-goal `session_id`. Sharing is safe because each goal's `bus.close()` closes the handle and `append` reopens it.
  - **Same managed layout + flags as batch:** `events/<repl-session-id>.jsonl`, `--log` honored, `latest.jsonl` pointer maintained only for the managed layout (swung after the cockpit exits; the file exists from launch, so it never dangles).
